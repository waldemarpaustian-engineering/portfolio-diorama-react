„Reaktivität" heißt: Wie kommt eine Änderung an deinen Daten auf den Bildschirm?
**React, Angular und Vue** beantworten diese Frage unterschiedlich — und alle
drei haben ihre Antwort zwischen 2020 und 2026 noch einmal geschärft. Das
Spannende: Sie nähern sich demselben Ziel (feinere, automatischere Updates), aber
jeder aus einer anderen Ecke.

![Reaktivität im Zeitverlauf](/blog/reactivity-react-vs-angular/timeline.svg)

## Die drei Grundmodelle in einem Satz

- **React**: *„UI = f(State)"*. Ändert sich State, **läuft die Komponente neu**,
  React vergleicht das Ergebnis (Virtual DOM) und patcht die Unterschiede.
  Grob-granular, dafür extrem simpel im Kopf.
- **Angular** (heute): **Signals** — ein reaktiver Wert, der weiß, **wer ihn
  liest**. Ändert er sich, updaten nur genau diese Stellen. Fein-granular, ohne
  Virtual DOM.
- **Vue**: **reaktive Daten per Proxy** — du mutierst den Wert ganz normal, Vue
  verfolgt automatisch, **was davon abhängt**. Fein-granular wie Angular, lange
  aber mit Virtual DOM gerendert — und genau den entfernt jetzt der Vapor Mode.

![Das Reaktivitäts-Spektrum: grob bis fein-granular](/blog/reactivity-react-vs-angular/models.svg)

## Geschichte: Wie React dahin kam

| Jahr | Meilenstein | Reaktivität |
|---|---|---|
| 2013 | React + Virtual DOM | `this.setState()` → Re-Render + Diff |
| 2019 | Hooks (16.8) | `useState`, manuelle Memoization |
| 2022 | Concurrent (18) | `useTransition`, Batching |
| 2024/25 | **React Compiler** | Memoization automatisch |

Reacts Modell ist im Kern seit 2013 gleich: Die Komponente ist eine Funktion, die
bei jeder Änderung **neu aufgerufen** wird. Unnötige Re-Renders hat man lange
**von Hand** mit `useMemo`/`useCallback` bekämpft — der **Compiler** nimmt einem
das ab, ohne das Modell zu ändern.

```jsx
// React: State ist ein roher Wert, Abgeleitetes rechnet man einfach aus
function Counter() {
  const [count, setCount] = useState(0);
  const doubled = count * 2; // kein computed nötig — Compiler memoisiert
  return <button onClick={() => setCount(count + 1)}>{count} / {doubled}</button>;
}
```

Bewusst hat React **keine** Signals eingeführt. Das Argument: die **Rohheit der
Daten**. `count` ist ein einfacher Wert — kein `.value`, kein Getter.

## Geschichte: Wie Angular dahin kam

| Jahr | Meilenstein | Reaktivität |
|---|---|---|
| 2010 | AngularJS | `$scope` + `$digest`-Dirty-Checking |
| 2016 | Angular 2 | **zone.js** + RxJS, Check über den Baum |
| 2023 | Angular 16/17 | **Signals** (stabil ab v17) |
| 2025 | Angular 21 | **zoneless** wird Default |

Angular ist von *„prüfe den ganzen Baum"* (zone.js) zu *„benachrichtige genau den
Knoten"* (Signals) gewandert:

```typescript
// Angular: Signals sind Getter-Funktionen — Lesen mit count()
@Component({ template: `{{ count() }} / {{ doubled() }}` })
export class CounterComponent {
  count = signal(0);
  doubled = computed(() => this.count() * 2);
  inc() { this.count.update((c) => c + 1); }
}
```

## Geschichte: Wie Vue dahin kam

| Jahr | Meilenstein | Reaktivität |
|---|---|---|
| 2014 | Vue 1/2 | `Object.defineProperty` (Getter/Setter) |
| 2020 | Vue 3 | **Proxy** + Composition API (`ref`, `reactive`) |
| 2024 | Vue 3.5 | Reaktivität neu gebaut (schneller, sparsamer) |
| 2026 | Vue 3.6 (Beta) | **Vapor Mode** + Alien Signals, **kein VDOM** |

Vue ist der interessante Sonderfall: Es war **von Anfang an fein-granular
reaktiv**. Schon Vue 2 verfolgte über Getter/Setter, welche Stelle welchen Wert
liest. Vue 3 ersetzte das durch **Proxies** — robuster (erkennt auch neu
hinzugefügte Properties) und schneller. `ref()` ist dabei praktisch ein Signal:

```vue
<script setup>
import { ref, computed } from 'vue'
const count = ref(0)                          // wie ein Signal
const doubled = computed(() => count.value * 2)
function inc() { count.value++ }              // im Script: .value
</script>

<template>
  <!-- im Template automatisch „entpackt": kein .value -->
  <button @click="inc">{{ count }} / {{ doubled }}</button>
</template>
```

Der Clou 2026: Mit **Vapor Mode** (Vue 3.6, Beta) kompiliert Vue Templates direkt
zu **fein-granularen DOM-Operationen — ganz ohne Virtual DOM**, ähnlich wie Solid
oder Angular. Damit holt Vue auf der Render-Seite das nach, was seine Reaktivität
schon lange konnte. (Stand Juni 2026: Beta, „production-evaluation ready", noch
nicht Default; Suspense bleibt vorerst auf dem VDOM-Pfad.)

## Direkter Vergleich

| | React (2026) | Angular (2026) | Vue (2026) |
|---|---|---|---|
| Reaktives Primitiv | roher Wert + `useState` | `signal()` (Getter) | `ref()` / `reactive()` (Proxy) |
| Abgeleiteter Wert | einfach ausrechnen | `computed()` | `computed()` |
| Seiteneffekt | `useEffect(fn, deps)` | `effect(fn)` | `watchEffect(fn)` |
| Wert lesen | roh: `count` | als Aufruf: `count()` | `.value` (Template: roh) |
| Dependencies | `deps`-Array, manuell | automatisch | automatisch |
| Render | Virtual DOM | kein VDOM | VDOM → **Vapor: kein VDOM** |
| Optimierung | **Compiler** | **Signals** | **Proxy-Tracking** |

Schön sichtbar wird die Philosophie an einer Zeile: In **React** ist `count` ein
roher Wert (Reaktivität implizit), in **Angular** liest du `count()` (explizit),
in **Vue** `count.value` — und im Vue-Template wird es automatisch entpackt.

## Die „große Konvergenz" — drei Wege, ein Ziel

Alle drei wollen 2026 dasselbe: feinere Updates ohne manuelle Memoization. Aber
die Wetten sind verschieden:

- **Vue & Angular** (und Solid, Svelte 5, Qwik, Preact) setzen auf **Signals /
  fein-granulare Reaktivität**. Der Wert weiß selbst, wer ihn liest — es muss
  nichts „diffen". Angular macht das explizit (`signal()`), Vue automatisch
  (Proxy). Beide entfernen inzwischen den Virtual DOM (zoneless bzw. Vapor).
- **React** ist das große Framework, das **bewusst beim Re-Render bleibt** und
  die Effizienz dem **Compiler** überlässt. Du schreibst „normalen" Code mit
  rohen Werten, die Optimierung passiert unsichtbar.

> **Kernaussage:** Die Signals-Bewegung wurde maßgeblich von **Vue** mitgeprägt —
> Vues reaktives Modell ist gewissermaßen der Urahn. Angular ist später dazu-
> gestoßen, und alle drei (plus Solid) arbeiten sogar an einem gemeinsamen
> **TC39-Signals-Standard**. React geht als einziges großes Framework den
> Compiler-Weg.

Und die Performance? In realen Apps meist ein **Unentschieden**. Fein-granular
(Vue/Angular/Solid) glänzt bei sehr vielen unabhängigen Werten und großen Listen.
Aber über Tempo entscheiden in der Praxis öfter **Datenfluss, Bundle-Größe und
Render-Grenzen** als das Reaktivitätsmodell selbst.

## Was das für dich heißt

- **React-Mindset**: „Die Funktion läuft neu." Mit dem Compiler kannst du die
  meisten `useMemo`/`useCallback` weglassen. Reaktivität ist **implizit**.
- **Angular-Mindset**: „Wer liest dieses Signal?" `computed` statt
  Neuberechnung, `effect` statt Subscriptions, Lesen mit `()`. **Explizit**.
- **Vue-Mindset**: „Mutiere den Wert, der Rest passiert von selbst." `ref`/
  `reactive`, `computed`, `watchEffect` — die Reaktivität ist **automatisch** und
  fühlt sich am ehesten wie „normales" JavaScript an.

## Fazit

Drei Frameworks, ein Ziel, drei Wetten. **Vue** war früh fein-granular reaktiv
und entfernt nun mit Vapor den letzten VDOM-Rest. **Angular** hat seine
Reaktivität komplett zu Signals umgebaut und zone.js verabschiedet. **React** hält
am bewährten Re-Render fest und glättet die Schmerzpunkte per Compiler. Keiner der
Wege ist „richtiger" — sie spiegeln unterschiedliche Wetten darauf, was Entwickler
mehr schätzen: **explizite Kontrolle**, **automatische Reaktivität** oder
**unsichtbare Optimierung**.
